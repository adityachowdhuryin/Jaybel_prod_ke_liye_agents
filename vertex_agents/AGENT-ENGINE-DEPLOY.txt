Vertex AI Agent Engine - two-agent layout (Google ADK)
======================================================

Project: gls-training-486405
Region:  us-central1
Model:   gemini-2.5-flash

Folders (pass these paths to adk deploy agent_engine):
  vertex_agents/cost_metrics_agent
  vertex_agents/pa_orchestrator_agent

Deploy order (fresh engines)
----------------------------
1) Deploy cost_metrics_agent first (BigQuery + clean_billing_view mode).
2) Copy the new cost engine ID/resource from deploy output or console.
3) Deploy pa_orchestrator_agent with COST_AGENT_QUERY_ENDPOINT pointing to the
   new cost engine.
4) Update local FastAPI proxy config:
   - ORCHESTRATOR_LOCAL_CHAT=0 (or unset)
   - ORCHESTRATOR_AGENT_ENGINE_RESOURCE=projects/.../reasoningEngines/<new orchestrator id>

Telemetry
---------
Use either deploy script (both include):
  --trace_to_cloud --otel_to_cloud

---------------------------

  .\scripts\deploy-agent-engine.ps1 -Agent cost -ForceNewEngine
  .\scripts\deploy-agent-engine.ps1 -Agent orchestrator -ForceNewEngine -CostAgentEngineId <NEW_COST_ENGINE_ID>

  ./scripts/deploy-agent-engine.sh cost --force-new-engine
  ./scripts/deploy-agent-engine.sh orchestrator --force-new-engine --cost-agent-engine-id <NEW_COST_ENGINE_ID>

Local stack check

After running .\scripts\start-all.ps1:
  .\scripts\check-local-stack.ps1

Next.js UI -> Vertex Agent Engine (every chat invokes the engine)
------------------------------------------------------------------
The browser cannot call reasoningEngines:query with your user credentials. Use the
FastAPI orchestrator as a proxy:

1. Deploy cost_metrics_agent then pa_orchestrator_agent (order above).
2. gcloud auth application-default login  (same account that can invoke the engine)
3. Export GOOGLE_CLOUD_PROJECT, GOOGLE_CLOUD_LOCATION, and:
     ORCHESTRATOR_AGENT_ENGINE_RESOURCE=projects/.../reasoningEngines/ID
   Ensure ORCHESTRATOR_LOCAL_CHAT is unset/0.
4. pip install -r agents/orchestrator/requirements.txt (includes google-cloud-aiplatform)
5. Run uvicorn on agents/orchestrator/main.py :8000
6. Keep frontend NEXT_PUBLIC_ORCHESTRATOR_URL=http://127.0.0.1:8000

GET /health shows agent_engine_chat_enabled: true when project + resource are set and local-chat override is off.
Agent Engine metrics in the console should increment when you send messages.

Evaluation + Memories tabs
--------------------------
- Memories tab population:
  - Agents must be memory-enabled (PreloadMemoryTool + after_agent_callback memory writes), then redeployed.
  - Run multi-turn scenarios against one or more deployed engines:
    - scripts/agent-engine-memory-smoke.py
  - Script now triggers explicit session->memory generation and can verify via memory search.
  - Example:
    - ./.venv/bin/python scripts/agent-engine-memory-smoke.py \
      --resource projects/.../reasoningEngines/<ORCHESTRATOR_ID> \
      --resource projects/.../reasoningEngines/<COST_ID> \
      --scenarios scripts/evals/memory_seed_cases.json \
      --verify-memory \
      --out logs/agent-engine-memory-seed-report.json

- Evaluation tab population:
  - Publish real eval runs through Vertex GenAI eval client:
    - scripts/agent-engine-create-eval.py --publish-to-vertex --gcs-dest gs://<bucket>/<path>
  - Cost-aware options for repeat runs:
    - `--max-cases N`, `--priority P0`, `--case-id <id>` to reduce run size.
    - `--minimal-vertex-eval` to publish a cheaper run with only `HALLUCINATION`.
  - Use `scripts/evals/agent_engine_eval_cases.json` for single-turn guardrails and
    `scripts/evals/agent_engine_multiturn_cases.json` for clarification-flow regressions.
  - Cheap committed smoke pack (5 P0 cases): `scripts/evals/eval_smoke_p0.json`
  - Clarification and error paths use the typed `COST_PAYLOAD_JSON:` line prefix; the
    eval harness matches `must_contain_any` / `must_not` against both final text and
    parsed JSON fields. Optional: `--turn-timeout-seconds` and `--turn-retries` for
    long orchestrator runs. Postgres (`DATABASE_URL`) is a **deprecated** fallback
    in `COST_DATA_SOURCE=auto` when BigQuery is unavailable; prefer BQ in production.
  - Versioned golden baseline exists at:
    - `scripts/evals/golden_dataset_v1.json`
    - `scripts/evals/golden_dataset_schema.json`
    - `scripts/evals/golden_dataset_readme.md`
  - Enforce deterministic quality gates:
    - `--fail-on-assertion`
    - `--fail-on-priority P0`
    - `--min-pass-rate 0.90` (seeding baseline) or `0.95` (release gate)
  - Include time-interpretation cases such as "this year" in the single-turn pack.
  - Example:
    - ./.venv/bin/python scripts/agent-engine-create-eval.py \
      --resource projects/.../reasoningEngines/<ORCHESTRATOR_ID> \
      --cases scripts/evals/agent_engine_eval_cases.json \
      --fail-on-assertion \
      --fail-on-priority P0 \
      --min-pass-rate 0.95 \
      --publish-to-vertex \
      --gcs-dest gs://<bucket>/agent-engine-evals \
      --display-name orchestrator-guardrail-eval

- One-command helper for both tabs (memory seeding + four eval runs: single-turn + multi-turn for each agent):
  - scripts/seed-agent-engine-observability.sh (macOS/Linux)
  - scripts/seed-agent-engine-observability.ps1 (Windows)
  - Optional cost controls (opt-in only; default behavior unchanged):
    - `SKIP_MEMORY_SMOKE=1`
    - `SKIP_VERTEX_PUBLISH=1`
    - `MINIMAL_VERTEX_EVAL=1`
  - Prereqs:
    - ORCHESTRATOR_AGENT_ENGINE_RESOURCE
    - COST_AGENT_ENGINE_RESOURCE
    - AGENT_ENGINE_EVAL_GCS_DEST

Online Monitors (continuous scoring)
------------------------------------
- Full guide: `docs/ONLINE_MONITORING_COST_AGENT.md`
- Configure cost agent online monitor only (50% sampling, 4 metrics) using:
  - macOS/Linux: `scripts/setup-agent-engine-online-monitor.sh`
  - Windows: `scripts/setup-agent-engine-online-monitor.ps1`
- The setup is idempotent: updates existing monitor with same display name.
- Metrics configured:
  - `HALLUCINATION`
  - `FINAL_RESPONSE_QUALITY`
  - `TOOL_USE_QUALITY`
  - `SAFETY`
- Verify local routing prerequisites:
  - `python scripts/check-online-monitor-prereqs.py`
- After chatting via local UI, online-monitor results usually surface in
  Agent Engine → Evaluation → Online Monitors within ~10-30 minutes.
